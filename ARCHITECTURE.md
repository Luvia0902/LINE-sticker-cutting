## 📊 資料流程

### 範例：LINE 貼圖切割任務

```
1. 使用者請求
   ↓
2. Agent 讀取 directives/example_directive.md
   ↓
3. Agent 理解需求：
   - 輸入：大圖 (4x5 配置)
   - 輸出：20 張獨立貼圖
   - 規格：370x320 px, PNG, <1MB
   ↓
4. Agent 編排執行順序：
   a. 驗證輸入 (validate_image.py) - 未來實作
   b. 切割圖片 (split_stickers.py) ✓ 已實作
   c. 壓縮優化 (compress_stickers.py) - 未來實作
   ↓
5. 依序執行腳本
   ↓
6. 收集結果，報告給使用者
```

## 🔄 自退火循環（Self-Annealing Loop）

當系統遇到錯誤時的自我改進流程：

```
錯誤發生
    ↓
分析錯誤訊息
    ↓
修復工具/腳本
    ↓
測試修復結果
    ↓
更新指令文檔
    ↓
系統變得更強大 ✓
```

### 範例場景

**問題**：API 速率限制導致失敗

**解決流程**：
1. 檢測到 `RateLimitError`
2. 研究 API 文檔，找到批次處理端點
3. 修改 `execution/api_caller.py` 使用批次處理
4. 添加重試機制和延遲
5. 測試確認可正常運作
6. 更新 `directives/api_usage.md` 記錄此限制
7. 系統現在能夠應對速率限制 ✓

## 🎯 核心設計原則

### 1. 關注點分離（Separation of Concerns）

| 層級 | 職責 | 特性 |
|------|------|------|
| 指令層 | 定義「做什麼」 | 自然語言，易於理解和修改 |
| 編排層 | 決定「如何做」 | 智慧路由，錯誤處理，學習改進 |
| 執行層 | 執行「實際操作」 | 確定性，可測試，快速可靠 |

### 2. 確定性 vs 機率性

**問題**：LLM 是機率性的，但業務邏輯需要確定性

**解決方案**：
- ✅ **指令層**：機率性（AI 可以理解自然語言）
- ✅ **編排層**：半機率性（AI 做決策，但遵循明確規則）
- ✅ **執行層**：完全確定性（Python 腳本保證一致性）

### 3. 可靠性計算

假設每一步成功率為 90%：

| 步驟數 | 傳統方式（AI 做所有事） | 三層架構方式 |
|--------|-------------------------|--------------|
| 1 步 | 90% | 98% (確定性腳本) |
| 3 步 | 73% (0.9³) | 94% (0.98³) |
| 5 步 | 59% (0.9⁵) | 90% (0.98⁵) |
| 10 步 | 35% (0.9¹⁰) | 82% (0.98¹⁰) |

**結論**：將複雜性轉移到確定性代碼，可大幅提升可靠性

## 🚀 工作流程

### 典型的任務執行流程

1. **使用者發起請求**
   ```bash
   python execution/agent_orchestrator.py --task split-stickers --input big_image.png
   ```

2. **編排器啟動**
   - 載入 `directives/example_directive.md`
   - 解析任務需求
   - 規劃執行步驟

3. **執行腳本**
   - 調用 `execution/split_stickers.py`
   - 傳遞參數
   - 監控執行狀態

4. **結果處理**
   - 收集輸出
   - 檢查錯誤
   - 記錄日誌

5. **反饋給使用者**
   - 顯示結果
   - 報告錯誤（如有）
   - 提供下一步建議

## 🔧 擴展系統

### 添加新任務

1. **創建指令** (`directives/new_task.md`)
   ```markdown
   # 新任務名稱
   
   ## 目標
   [說明目標]
   
   ## 輸入
   [定義輸入]
   
   ## 使用工具/腳本
   [列出腳本]
   
   ## 輸出
   [說明輸出]
   ```

2. **創建執行腳本** (`execution/new_script.py`)
   ```python
   # 使用標準模板
   # 實作具體邏輯
   ```

3. **在編排器中註冊**
   ```python
   # 在 agent_orchestrator.py 添加新任務
   if args.task == 'new-task':
       # 實作任務編排邏輯
   ```

4. **測試**
   ```bash
   python execution/agent_orchestrator.py --task new-task --input test.txt
   ```

5. **文檔化**
   - 更新 README.md
   - 記錄已知限制
   - 添加範例

## 📈 最佳實踐

### ✅ DO（應該做）

- ✅ 先檢查現有工具，避免重複造輪子
- ✅ 將錯誤當作學習機會，更新指令
- ✅ 保持腳本簡單、專注、可測試
- ✅ 使用清晰的命名和註解
- ✅ 將中間文件放在 `.tmp/`
- ✅ 最終交付物存雲端（Google Sheets/Slides）

### ❌ DON'T（不應該做）

- ❌ 在 AI 層做確定性操作（用腳本代替）
- ❌ 未經測試就部署新腳本
- ❌ 忽略錯誤訊息和警告
- ❌ 提交 `.env`、`credentials.json` 等敏感文件
- ❌ 在生產環境中使用未記錄的 API
- ❌ 讓臨時文件累積（定期清理 `.tmp/`）
